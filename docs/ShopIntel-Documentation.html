1. Executive Overview

ShopIntel is an autonomous AI agent for Shopify merchants, built as a full-stack Next.js application. It combines a real-time analytics dashboard with a natural-language chat interface powered by GPT-5.4. The AI agent doesn't just answer questions — it proactively analyzes store data, predicts trends, generates production-grade Shopify Liquid code, and deploys it directly to live themes.

Key Capabilities

  • Natural Language Querying — Ask "How are sales?" and get interactive charts, KPIs, and forecasts
  • Generative UI — AI selects and renders the right React component (charts, tables, grids) in real-time
  • Trend Forecasting — WoW/MoM growth, anomaly detection, product-level trends
  • Liquid Code Generation — AI writes complete Shopify sections with preview
  • One-Click Deploy — Push generated code directly to the live Shopify theme
  • Persistent Conversations — Chat history saved to PostgreSQL, restored on refresh

Architecture Summary

2.2 Request Flow — AI Chat

  1. User types message in chat input
  2. usePersistedChat hook creates/loads conversation in PostgreSQL
  3. Message sent to /api/chat as streaming POST request
  4. Proxy middleware verifies JWT session token
  5. API converts messages, calls streamText() with GPT-5.4 + tool definitions
  6. Model decides which tools to call (getAnalytics, getProducts, etc.)
  7. Tools execute Shopify GraphQL queries, return structured data
  8. Results stream back as UI message chunks
  9. ToolResultRenderer maps tool names to React components (charts, tables, grids)
  10. Components render in real-time as data arrives
  11. On completion, assistant message saved to PostgreSQL

4. Authentication & Security

4.1 JWT Authentication

Authentication is implemented via JSON Web Tokens (JWT) using the jose library. Sessions are stateless — the JWT contains the full user identity and is signed with HS256.

Session Lifecycle

  1. Login: User submits email + password to /api/auth/login
  2. Verification: Password checked against bcryptjs hash (or legacy SHA256 with auto-upgrade)
  3. Token Creation: JWT signed with JWT_SECRET, containing {id, email, name, role}
  4. Cookie: Token stored as session cookie with security flags
  5. Verification: Every request verified by proxy middleware
  6. Expiration: Tokens expire after 24 hours

Cookie Configuration

{
  httpOnly: true,       // Not accessible via JavaScript
  secure: true,         // HTTPS only
  sameSite: "strict",   // No cross-site sending
  maxAge: 86400,        // 24 hours
  path: "/"             // Available site-wide
}

Session Interface

interface SessionUser {
  id: string;       // CUID from database
  email: string;    // User email
  name: string;     // Display name
  role: "admin" | "client";
}

4.2 Password Hashing

Passwords are hashed with bcryptjs using 12 salt rounds. The system supports legacy SHA256 hashes and auto-upgrades them on successful login.

// Hashing (on registration)
const hash = await bcrypt.hash(password, 12);

// Verification (on login)
const valid = await bcrypt.compare(password, storedHash);

// Legacy migration: if hash doesn't start with "$2", try SHA256
// On success, auto-upgrade to bcryptjs hash

4.3 Middleware & Security Headers

File: src/proxy.ts — Next.js 16 proxy middleware that intercepts every request.

Authentication Flow

  │

5. Database & Railway

5.1 Schema Design

Three models with simple relational structure. Messages cascade-delete when a conversation is removed.

┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ User │ │ Conversation │ │ Message │ ├──────────────┤ ├──────────────────┤ ├──────────────┤ │ id (PK) │──┐ │ id (PK) │──┐ │ id (PK) │ │ email (UQ) │ │ │ title │ │ │ role │ │ password │ └───>│ userId (FK) │ └───>│ conversationId│ │ name │ │ createdAt │ │ content │ │ role │ │ updatedAt │ │ createdAt │ │ createdAt │ └──────────────────┘ └──────────────┘ └──────────────┘ 1:many 1:many (cascade delete)

Prisma Schema

model User {
  id            String         @id @default(cuid())
  email         String         @unique
  password      String
  name          String
  role          String         @default("client")
  createdAt     DateTime       @default(now())
  conversations Conversation[]
}

model Conversation {
  id        String    @id @default(cuid())
  title     String    @default("New Conversation")
  userId    String
  user      User      @relation(fields: [userId], references: [id])
  messages  Message[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

model Message {
  id             String       @id @default(cuid())
  conversationId String
  conversation   Conversation @relation(fields: [conversationId],
                               references: [id], onDelete: Cascade)
  role           String       // "user" | "assistant"
  content        String       // JSON-serialized message parts
  createdAt      DateTime     @default(now())
}

5.2 Prisma ORM Configuration

Prisma 7.5 with the @prisma/adapter-pg for native PostgreSQL connection pooling. The client is generated to src/generated/prisma/ and instantiated as a singleton.

// prisma.config.ts
import path from "node:path";
import { PrismaPg } from "@prisma/adapter-pg";
import { defineConfig } from "prisma/config";

export default defineConfig({
  earlyAccess: true,
  schema: path.join(__dirname, "prisma/schema.prisma"),
  migrate: { adapter: () => new PrismaPg({ connectionString: process.env.DATABASE_URL! }) },
});

// src/lib/db.ts — Singleton pattern
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };
export const prisma = globalForPrisma.prisma ?? new PrismaClient({
  adapter: new PrismaPg({ connectionString: process.env.DATABASE_URL! }),
});
if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;

5.3 Railway Configuration

Railway PostgreSQL Instance
  • Provider: Railway (railway.app)
  • Engine: PostgreSQL 16
  • Connection: Via proxy URL (ballast.proxy.rlwy.net:24687)
  • Auth: Username/password in connection string
  • Pooling: Handled by Prisma PrismaPg adapter
  • Backups: Railway automatic daily backups

Connection String Format:

postgresql://<user>:<password>@ballast.proxy.rlwy.net:24687/railway
Important: The DATABASE_URL must be set in both .env.local (local dev) and Vercel environment variables (production). Railway provides the connection string in the dashboard under your PostgreSQL service settings.

7. AI Agent System

7.1 System Prompt

The AI agent is configured as a sharp, strategic commerce advisor with specific behavioral rules:

  • Proactive: Surface anomalies and opportunities unprompted
  • Tool Chaining: "How's business?" triggers getAnalytics + forecastTrends + getInventory
  • Action-Oriented: Generate Liquid code AND offer to deploy immediately
  • Predictive: Always frame data with growth trends and forecasts
  • Concise: Let the UI components do the heavy lifting — text should be scannable

Liquid Generation Design Direction

When generating Shopify Liquid sections, the AI follows a luxury clean beauty aesthetic (Kosas-inspired):

  • Typography: Founders Grotesk headings, Roboto body, generous letter-spacing
  • Palette: Black/white primary, dusty rose (#D33167) accent
  • Spacing: 80-100px vertical padding, 24-32px grid gaps
  • Motion: Subtle 0.3s transitions, hover scale 1.02-1.05
  • Buttons: Solid black, uppercase, minimal radius, rose hover

7.3 Streaming & Generative UI

The chat uses the Vercel AI SDK's streaming protocol. As the model generates text and tool calls, they're streamed to the client as UI message chunks. The ToolResultRenderer component maps each tool's output to the appropriate React component:

// Tool name → Component mapping
switch (toolName) {
  case "getProducts":   return <ProductGrid products={result.products} />
  case "getOrders":     return <OrdersTable orders={result.orders} />
  case "getCustomers":  return <CustomerList customers={result.customers} />
  case "getAnalytics":  return <KPICards ... /> + <RevenueChart ... />
  case "getInventory":  return <InventoryTable ... />
  case "forecastTrends": return <TrendForecast data={result} />
  case "generateLiquid": return <LiquidPreview code={result.code} ... />
  case "deploySection":  return <DeployAlert ... />
}

8.3 Analytics Engine

Analytics are computed server-side from raw order data (not Shopify's analytics API):

  • Revenue: Sum of totalPriceSet.shopMoney.amount across all orders
  • AOV: Total revenue / order count
  • Top Products: Aggregated from line items, sorted by revenue (top 10)
  • Revenue Timeline: Daily aggregation from order createdAt
  • Status Breakdown: Count of orders per displayFulfillmentStatus

8.4 Trend Forecasting

The forecastTrends() function provides predictive analytics:

  • WoW Growth: Compares last 7 days revenue vs prior 7 days
  • MoM Growth: Compares last 30 days vs prior 30 days
  • Product Trends: Top 5 growing + top 5 declining products (by revenue change)
  • Anomaly Detection: Flags daily revenue >2 standard deviations from 30-day mean
  • Statistics: 30-day mean and standard deviation

8.5 Theme Deployment

The deploySection tool pushes Liquid code directly to the active Shopify theme:

  1. Fetches the main theme ID via themes(first:1, roles:MAIN)
  2. Upserts the file at sections/{sectionName}.liquid
  3. Returns success status with filename
Production Warning: The deploy tool writes directly to the live theme. In production, consider adding a staging theme workflow or confirmation step before deploying to the main theme.

9. Frontend & Components

9.1 Dashboard Layout

The main dashboard (src/components/dashboard.tsx) is a server-rendered page that fetches all initial data in parallel, then renders an interactive client component with:

  • Header: Logo, branding, quick-action suggestion buttons (admin only), user info, logout
  • KPI Section: 4-column metric cards (revenue, orders, AOV, low stock)
  • Charts Row: Asymmetric 12-col grid — Revenue (5-col), Top Products (4-col), Status (3-col)

  • Inventory Table: Full-width with low-stock highlighting, admin-only

9.3 Chat Interface

Two chat implementations exist:

  1. Dashboard Overlay (dashboard.tsx) — Floating card on dashboard, minimizable, with sticky input bar
  2. Full Chat (chat/chat-interface.tsx) — Standalone page with conversation sidebar, history management

Both use the usePersistedChat hook which wraps useChat from the Vercel AI SDK with automatic conversation creation, message persistence to PostgreSQL, and conversation restore on page load.

9.4 Liquid Preview System

The LiquidPreview component renders generated Liquid code in two views:

  • Code Tab: Full Liquid source with copy button
  • Preview Tab: Sandboxed iframe with product data injected, rendering the HTML/CSS output
  • Deploy Button: One-click push to Shopify theme via deploySection tool
